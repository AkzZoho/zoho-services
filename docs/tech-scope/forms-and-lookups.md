# Forms & Lookups — Project Rules

> **Status:** Canonical · Applies to every Tech Scope generated by this tool
> **Owner:** Tech Scope module
> **Last updated:** 2025

This document captures the **non-negotiable rules** the Tech Scope tool follows
when producing the Forms specification (Step 2 — Data Model) for a Zoho
Creator application. Any UI, exporter, or AI-assisted generator that emits
a forms list **must** comply.

---

## 1. Rule: Every Dropdown is its own Form (Lookup)

**All dropdown-style fields MUST be implemented as standalone Lookup forms.**

This includes:

- `Dropdown` (single-select picklist)
- `Radio` (single-select via radio buttons)
- `Multi-Select` (multi-value picklist)
- `CheckBox` (multi-value checkbox group)

### Why

In Creator, a static dropdown bakes its values into the form definition. To
add a new value later, the form must be **edited and redeployed**. By
promoting the dropdown to its own master form, business users can add new
values from the **record interface at runtime — no code change required.**

### How (canonical pattern)

```
# ❌ DO NOT: static dropdown
Form: Employee
  • Department  — Dropdown  — values: [Sales, Engineering, HR]

# ✅ DO: standalone lookup master + Single Select Lookup
Form: Department_Master       (lookup form — built FIRST)
  • Name  — Single Line  — Mandatory · Unique
  • Code  — Single Line  — Optional  · Unique

Form: Employee
  • Department  — Single Select Lookup → Department_Master.Name  — Mandatory
```

### Naming convention

- Lookup master form: `<FieldName>_Master` (e.g. `Status_Master`, `Country_Master`)
- Mandatory fields on every master: `Name` (Single Line, Unique) — at minimum
- Optional fields commonly added: `Code`, `Order`, `Active` (Decision Box)

---

## 2. Lookup cardinality

| User intent                          | Field type to use         |
|--------------------------------------|---------------------------|
| Pick exactly one value               | **Single Select Lookup**  |
| Pick one or more values              | **Multi-Select Lookup**   |
| Embed multiple full child records    | **Subform**               |

A `Multi-Select Lookup` stores an array of foreign keys; a `Subform` stores
embedded records — choose based on whether the related rows have their own
attributes the parent needs to capture inline.

---

## 3. Form Creation Order (deterministic)

When listing forms in the Tech Scope, the tool **must** present them in
dependency order:

1. **Tier 0** — Lookup master forms with no inbound dependencies
   (e.g. `Status_Master`, `Country_Master`, `Department_Master`)
2. **Tier 1** — Forms whose only lookups point to Tier 0
3. **Tier N** — Forms whose lookups point to Tier 0..N-1
4. Any cycle remnants are appended at the end (these need designer review)

The `FormsView.jsx` component uses Kahn's algorithm with alphabetical
tie-breaking so the order is **stable and reproducible** across runs.

---

## 4. Mandatory-field heuristics

A field is marked **Mandatory** in the spec when **any** of the following
hold (in priority order):

1. The BRD explicitly says "required", "mandatory", or marks the field with `*`
2. The field is the natural primary identifier (`ID`, `Code`, `Name` on a master)
3. The field is a `Single Select Lookup` to a Tier-0 master form on a
   transactional form (you almost always want a non-null FK)
4. The field is `Auto Number` (Creator generates it, but it's effectively required)

Optional fields default to `required: false`.

## 5. Worked sample

```
# 1. Lookup masters (build first — no inbound deps)
Form: Department_Master
  • Name           — Single Line   — Mandatory · Unique
  • Code           — Single Line   — Mandatory · Unique

Form: Status_Master
  • Name           — Single Line   — Mandatory · Unique
  • Order          — Number        — Optional

Form: Skill_Master
  • Name           — Single Line   — Mandatory · Unique

# 2. Transactional forms (build after their lookups)
Form: Employee
  • Employee_ID    — Auto Number   — Mandatory · Unique
  • Full_Name      — Name          — Mandatory
  • Email          — Email         — Mandatory · Unique
  • Department     — Single Select Lookup → Department_Master    — Mandatory
  • Skills         — Multi-Select Lookup → Skill_Master
  • Status         — Single Select Lookup → Status_Master        — Mandatory
  • Joined_On      — Date          — Mandatory
```

## 7. Compliance check

Before marking a Tech Scope "complete", verify:

- [ ] Every `Dropdown`, `Radio`, `Multi-Select`, `CheckBox` field has a
      corresponding `_Master` form OR is explicitly justified as static
      (e.g. `Yes/No` toggles → use `Decision Box`, not `Dropdown`)
- [ ] Every lookup field's `target` form exists in the scope's `forms[]`
- [ ] Every master form has a `Name` field marked `Unique` + `Mandatory`
- [ ] Forms list is presented in topological-sort order
